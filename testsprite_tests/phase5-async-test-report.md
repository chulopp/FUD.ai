## 4️⃣ Key Gaps / Risks

### GAP-01: TestSprite PRD Auto-Generation Lag
TestSprite auto-generates test cases from the cached PRD which described the old synchronous 200 response. Until the PRD is updated in the TestSprite project to reflect the 202 async response, TC001/TC003/TC004 will always fail.

**Recommendation**: Update the TestSprite project PRD to include:
- `POST /api/agent` → `202 Accepted` with `job_id` + `poll_url`
- `GET /api/agent/<job_id>` → polling endpoint spec
- Then regenerate test cases.

### GAP-02: Proxy Timeout Fully Eliminated for Initial Request
The original 30-second proxy timeout issue (Phase 4 finding) is **architecturally solved**. POST now returns in < 1s (202). The pipeline runs in the background. Clients that poll are not subject to the proxy timeout on the POST call.

**Remaining risk**: The GET polling requests are short (< 1s each), so they are also proxy-safe. The pipeline itself may take 30-40s, but no single HTTP connection is held open during that time.

### GAP-03: waitUntil on Local Node.js Dev Server
`waitUntil()` is the Vercel/Edge standard. On local `npm run start` (standalone Node.js), the fallback `backgroundPipeline.catch(console.error)` is used instead. The background task continues via the Node.js event loop. In production Vercel deployments, `req.waitUntil()` will be used automatically.

### GAP-04: Frontend Client Not Yet Updated
The frontend UI still submits a POST and waits for a synchronous 200 response. It needs to be updated to:
1. Accept 202 + `job_id`
2. Implement a polling loop against `/api/agent/<job_id>`
3. Show a "Analyzing..." state while polling
